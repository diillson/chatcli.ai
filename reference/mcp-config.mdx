### Hot-Reload e TolerĂ˘ncia a Erros

O watcher do `mcp_servers.json` reconcilia o estado vivo com o que estĂĄ em disco usando `fsnotify`. Eventos de `Create`, `Write`, `Rename` e `Remove` sĂŁo debounced em 400 ms para evitar reload mid-write quando editores reescrevem via rename.

| Evento em disco | Resultado |
|:----------------|:----------|
| Adicionar servidor novo | Inicia o servidor, descobre tools |
| Remover servidor da config | Para o processo, drop das tools |
| Mudar `command`/`args`/`env` | Para e reinicia com nova config |
| `enabled: false` | Equivale a remover (servidor para) |
| Arquivo deletado | Para todos os servidores |
| **0 bytes ou JSON malformado** | Loga warning, **nĂŁo aborta**; corrija e salve para recarregar |

<Tip>
Salvar um JSON invĂĄlido **nĂŁo derruba o ChatCLI**: o manager continua registrado e o watcher segue ativo. VocĂŞ corrige o arquivo no editor, salva, e o prĂłximo evento dispara `Reload` normalmente. Ătil quando vocĂŞ estĂĄ editando ao vivo durante uma sessĂŁo.

## Formato


## VariĂĄveis de Ambiente (`env`)

O campo `env` aceita um objeto chave-valor que ĂŠ **mesclado** com o ambiente do processo pai (`os.Environ()`). Duas regras importantes:

### 1. HeranĂ§a do PATH e caches

O processo MCP **herda** todo o ambiente do ChatCLI antes de aplicar os valores de `env`. Sem essa heranĂ§a, launchers como `npx`, `uvx`, `docker` e `pipx` nĂŁo encontrariam seus binĂĄrios nem seus caches por usuĂĄrio.

```json

  "transport": "stdio",
  "command": "npx",
  "args": ["-y", "@anthropic/mcp-server-filesystem", "/workspace"],
  "env": {
    "FS_LOG_LEVEL": "debug"
  }
}
```

Aqui o servidor filho recebe **PATH + HOME + FS_LOG_LEVEL=debug** â nĂŁo apenas `FS_LOG_LEVEL` isolado. Se a chave do `env` colidir com o ambiente herdado, **o valor do `env` ganha** (semĂ˘ntica do Unix: Ăşltima atribuiĂ§ĂŁo no `cmd.Env` prevalece).

### 2. ExpansĂŁo `${VAR}` / `$VAR`

Valores em `env` passam por `os.Expand` contra o ambiente pai antes de ir pro processo filho. Isso permite manter **secrets fora do JSON** â em variĂĄveis de shell, ou em arquivos `.env` carregados via `source`/`direnv`/`mise`/`asdf`.

```json

  "name": "github",
  "transport": "stdio",
  "command": "npx",
  "args": ["-y", "@anthropic/mcp-server-github"],
  "env": {
    "GITHUB_TOKEN": "${GH_TOKEN}",
    "DEBUG": "$NODE_DEBUG"
  }
}
```

Se o shell tem `GH_TOKEN=ghp_xxx` exportado, o servidor MCP recebe `GITHUB_TOKEN=ghp_xxx`. Se a variĂĄvel referenciada **nĂŁo existe**, o resultado ĂŠ **string vazia** â mesmo comportamento do shell. FaĂ§a lint do seu shell antes de assumir que o token chegou.

<Info>
Use sempre `${VAR}` em vez de literais (`"GITHUB_TOKEN": "ghp_xxx"`). Tokens em plaintext no JSON acabam em git history, em logs, em backups e em screenshots. A expansĂŁo `${VAR}` ĂŠ a forma idiomĂĄtica â paridade com Claude Desktop e OpenCode.
</Info>

<Warning>
A expansĂŁo acontece **uma vez no spawn do processo**. Se vocĂŞ mudar `GH_TOKEN` no shell durante uma sessĂŁo, o servidor MCP que jĂĄ estava rodando continua com o valor antigo. Use `/mcp restart <nome>` para forĂ§ar a re-resoluĂ§ĂŁo.

### PadrĂľes recomendados

## Auto-aprovaĂ§ĂŁo e Trust

ChatCLI's tool-execution pipeline tem um audit-log que registra toda invocaĂ§ĂŁo MCP coberta por `autoApprove`, `alwaysAllow` ou `trust`. O helper `Manager.ShouldAutoApprove(toolName)` consultado pelo pipeline jĂĄ estĂĄ pronto pro futuro gate de aprovaĂ§ĂŁo interativa no coder mode.

### `autoApprove` / `alwaysAllow`

Listas de tool names que dispensam aprovaĂ§ĂŁo. Aceitam `"*"` (qualquer tool do server). Os nomes podem ser usados com ou sem o prefixo `mcp_` â `"read_file"` e `"mcp_read_file"` casam.

```json

  "transport": "stdio",
  "command": "npx",
  "args": ["-y", "@modelcontextprotocol/server-filesystem", "/workspace"],
  "enabled": true,
  "autoApprove": ["read_file", "list_directory"]
}
```

`alwaysAllow` ĂŠ alias popularizado pelo Cline â chatcli funde as duas listas no mesmo set runtime, entĂŁo copy-paste de configs do Cline funciona sem renomear.

### `trust: true`

Auto-aprova **toda** tool do server, sem precisar listar uma por uma:

```json
{
  "name": "my-trusted-tools",
  "transport": "stdio",
  "command": "/usr/local/bin/my-mcp",
  "enabled": true,
  "trust": true
}
```

<Warning>
`trust: true` ĂŠ destinado a servidores que o operador **separadamente verificou** (cĂłdigo auditado, container imutĂĄvel, etc.). ChatCLI emite um warning no startup toda vez que conecta com `trust: true` pra que a escolha fique visĂ­vel em logs â uma config trust commitada por engano nĂŁo passa silenciosa.
</Warning>

### Audit log

Toda chamada auto-aprovada gera um entry info-level:

```

```

Grep no log do chatcli (`CHATCLI_LOG_LEVEL=info` ou maior) para auditar tudo que passou pelo bypass.

## Timeouts

PadrĂľes cobrem o caso comum (`npx -y` cold start). Override quando o servidor demora mais que isso por desenho.

### `timeout` (segundos)

Cap por chamada RPC. PadrĂŁo **60s**. Aplica-se tanto a `tools/call` quanto a `tools/list`.

```json

  "transport": "sse",
  "url": "https://proxy.internal/sse",
```

### `initTimeout` (segundos)

```json
  "url": "https://bedrock-proxy.internal/sse",
  "enabled": true,

```

<Info>
SSE: o `http.Client.Timeout` ĂŠ setado para `max(timeout, initTimeout)` pra que um `timeout` curto nĂŁo mate o GET de SSE (que fica aberto pelo tempo da conexĂŁo).

## Headers HTTP e autenticaĂ§ĂŁo â SSE

### `headers`

Headers extra anexados a **todo** GET do stream SSE e **todo** POST de mensagens JSON-RPC:

```json

  "name": "internal-mcp",
  "transport": "sse",
  "url": "https://mcp.internal/sse",
  "enabled": true,
  "headers": {
    "X-Tenant-Id": "acme",
    "X-Trace-Id": "${TRACE_ID}"
  }
}
```

Values passam por `${VAR}` expansion (mesmo lookup do `env`).

### AutenticaĂ§ĂŁo SSE (`auth`)

Bloco tipado com trĂŞs modos:

#### `bearer`

```json
{
  "auth": {
    "type": "bearer",
    "token": "${MY_TOKEN}"
  }
}
```

Resulta em `Authorization: Bearer <token>`.

#### `basic`

```json
{
  "auth": {
    "type": "basic",
    "username": "${MY_USER}",
    "password": "${MY_PASS}"
  }
}
```

Resulta em `Authorization: Basic base64(user:pass)`.

#### `header` (custom)

```json
{
  "auth": {
    "type": "header",
    "header": "X-API-Key",
    "token": "${MY_TOKEN}"
  }
}
```

Resulta em `X-API-Key: <token>`. Quando `header` ĂŠ omitido, o default ĂŠ `X-API-Key`.

<Info>
**Salvaguardas**: empty `type` ĂŠ no-op mesmo com `token`/`username` populados (evita meio-config vazando credencial). Token cuja env var nĂŁo existe **suprime o header inteiro** em vez de mandar `Authorization: Bearer ` vazio (que passaria por um auth check ingĂŞnuo no server).
</Info>

<Warning>
`Authorization`/headers customizados sĂŁo re-aplicados a cada request, entĂŁo **rotacionar a env var em runtime** jĂĄ produz a prĂłxima chamada com o token novo â sem `/mcp restart`.
