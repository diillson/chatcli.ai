Após o teste de caos executado em **23/05/2026** sobre a v1.122.3 (kind cluster `nginx-cluster`), 5 gaps foram identificados e corrigidos. Esta página agrega o que mudou em comportamento, schema e ferramentas.

<Warning>
**Compatibilidade**: a maioria das mudanças é aditiva. As únicas mudanças que afetam usuários existentes:

- `Instance.spec.aiops.dedupTTLMinutes` default mudou de **60 para 30** minutos
- Novo estado `Contained` no enum de `Issue.status.state` — operadores customizados que tratam Issue states explicitamente precisam reconhecer este valor
- Action de remediação que silenciou o workload via containment **não declara mais o Issue como Resolved** (transita para `Contained`)
</Warning>

## GAP-01 — Desconexão AIInsight ↔ RemediationPlan (Alta)

### Sintoma
O AIInsight concluía corretamente "Scaling to 0 is the only correct containment action", mas o `RemediationPlan` gerado tentava `ExecDiagnostic` com um comando fora da allowlist. O Issue era marcado como Resolved por exaustão de `maxRemediationAttempts`, sem remediação real.

### Fix
- O prompt do `AgenticStep` agora recebe o AIInsight como **PRIMARY GUIDANCE** (análise + recomendações + suggested_actions + confidence), com instrução explícita: "se desviar da primeira ação sugerida, populate `divergence_reason`".
- O servidor (handler_analysis.go) compara `next_action` com `insight_suggested_actions[0]`. Se divergir, marca `diverges_from_insight=true` na resposta.
- O operator (remediation_controller.go) **recusa** executar uma ação divergente sem justificativa: a step é registrada como `REJECTED` no `AgenticHistory` e o controller pede uma nova proposta.
- Validação adicional no `AnalyzeIssue`: detecta quando a prose menciona "scale to 0/containment/stop the bleeding/unrecoverable" mas a primeira ação é `ExecDiagnostic` — loga warning para auditoria.

### Schema affected
- **proto**: `AgenticStepRequest` ganha `insight_analysis`, `insight_confidence`, `insight_recommendations`, `insight_suggested_actions`. `AgenticStepResponse` ganha `diverges_from_insight` e `divergence_reason`.

## GAP-02 — Dedup SHA256 ignora recriação de recursos (Alta)

### Sintoma
Deployment `chaos-crashloop` deletado e recriado com o mesmo nome ficava invisível para o AIOps por até 120 minutos (TTL antigo). Padrão comum em GitOps e rollbacks.

### Fix (3 camadas)

1. **UID-aware hashing** (correção estrutural): o `alertHash` agora inclui o UID do recurso K8s, obtido via lookup live. Recurso recriado → novo UID → hash diferente → novo Anomaly. Sentinela `uid:missing` quando o recurso não existe mais (ainda distinto de qualquer UID real).
2. **TTL default reduzido**: `Instance.spec.aiops.dedupTTLMinutes` cai de **60 → 30 minutos** (range `5-1440` mantido).
3. **InvalidateDedupForResource refatorado**: o dedup map agora carrega `(deployment, namespace)` com cada hash, permitindo invalidação por resource ref sem precisar recomputar hashes (que dependeriam do UID, que o caller não tem).

### RBAC requisito
O operator precisa de `get` em `apps/deployments,statefulsets,daemonsets,replicasets`, `batch/jobs,cronjobs`, `nodes`. Todos já cobertos pelo `chatcli-watcher` ClusterRole pré-existente.

## GAP-03 — Estado `Contained` distinto de `Resolved` (Média)

### Sintoma
`ScaleDeployment replicas=0 containment=true` silenciava o workload mas marcava o Issue como **Resolved**, mascarando indisponibilidade real. PostMortem era gerado normalmente, sem indicação de que serviço estava morto.

### Fix
Novo estado `Contained` na máquina de estados, com semântica explícita "serviço silenciado, ação humana pendente":

- `IssueState` ganha `Contained` (semi-terminal — auto-resolve quando humano restaura replicas)
- `handleContained` reconcile path usa `isResourceRestored`: requer `desired > 0 AND ready >= desired AND unavailable == 0` (mais estrito que `isResourceHealthy`, que aceitaria `0/0`)
- `PostMortemSpec` ganha `requiresHumanAction bool` e `requiredAction string`
- `PostMortemReconciler` **reverte** PostMortems marcados como `Closed` indevidamente de volta para `Open` enquanto a anotação `aiops.chatcli.io/human-action-acknowledged` não estiver setada
- `NotificationReconciler` dispara automaticamente em `state == Contained` se a `NotificationPolicy` incluir esse estado no filtro
- `AuditRecorder.RecordIssueContained` registra o evento em severity `warning` (NÃO info — Issue não está resolvido)

### Como reconhecer a ação humana
```bash
# REST API
POST /api/v1/postmortems/<name>/ack-human-action?namespace=<ns>
Body: {"acknowledgedBy":"sre-team","note":"rolled back to v1.2.3"}

# kubectl
kubectl annotate postmortem <name> -n <ns> \
  aiops.chatcli.io/human-action-acknowledged=true

# Dashboard
Botão "Ack Human Action" (roxo) na linha do PostMortem
```

### REST API affected
- `IncidentItem` ganha `requiresHumanAction bool`
- `PostMortemItem` ganha `requiresHumanAction`, `requiredAction`
- `AnalyticsSummary` ganha `containedIssues`, `postMortemsRequiringHumanAction`
- Nova rota: `POST /api/v1/postmortems/{name}/ack-human-action`

## GAP-04 — Contexto chaos sem comportamento diferenciado (Baixa-Média)

### Sintoma
IA identificava corretamente o cenário como chaos engineering, mas o pipeline executava idêntico a um incidente real. PostMortems de chaos drills poluíam o histórico de produção e MTTD/MTTR.

### Fix
`AnomalyReconciler` chama `FindActiveChaosExperiment(target)` ao criar Issues. Se há `ChaosExperiment` em estado `Running` (ou nos 2min após `Completed`/`Aborted` — janela de recuperação) no mesmo namespace+kind+name, o Issue recebe labels:

```yaml
platform.chatcli.io/source: chaos-experiment
platform.chatcli.io/chaos-experiment: <nome>
```

Esses labels alteram:
- Escalação L1→L2 suprimida (`NotificationReconciler` skipa `initiateEscalation`)
- Métrica `issueResolutionDuration` (MTTR de produção) excluída
- `analytics/summary.chaosInducedIssues` contado separadamente
- Labels propagam para o PostMortem
- Dashboard tem filtro dedicado (default: ocultar chaos drills)

Veja [Chaos Engineering — Correlação Automática](/features/aiops/chaos-engineering#correlacao-automatica-gap-04-fix-2026-05-23) para detalhes.

## GAP-05 — Visibilidade da allowlist `ExecDiagnostic` (Baixa)

### Sintoma
Allowlist padrão (~90 comandos) não incluía portas além das padrão (8080, 9090, 15000, 15020, 15021, 6060). Workload em `localhost:5678/health` ficava com diagnóstico cego, e o gap só era descoberto após observar falha no `RemediationPlan`. Não havia auditoria do que estava efetivamente carregado.

### Fix
- Função pública `controllers.GetDiagnosticAllowlistSummary()` retorna `{DefaultCount, CustomCount, TotalCount, Custom[]}`.
- `operator/main.go` loga no startup:
  ```
  Effective ExecDiagnostic allowlist loaded total=92 defaults=90 custom_count=2 custom=[...]
  ```
- `deploy/helm/chatcli-operator/templates/NOTES.txt` mostra um checklist pós-install com instruções explícitas para revisar a allowlist.
- Variável: `CHATCLI_ALLOWED_DIAGNOSTIC_COMMANDS` (comma-separated) continua sendo a forma de estender.

### Como auditar a allowlist efetiva
```bash
kubectl -n <ns> logs deploy/chatcli-operator \
  | grep "Effective ExecDiagnostic allowlist loaded"
```

## GAP-06 — CRDs não atualizados no `helm upgrade` (Alta, regressão pós-1.122.3)

### Sintoma
Após upgrade do chart, todo Issue que devia transitar para `Contained` ficava preso em `Remediating` até timeout (depois `Escalated`). Logs do operator:

```
"Issue transitioned to Contained (requires human action)"
... Reconciler error
Issue.platform.chatcli.io is invalid: state: Unsupported value: "Contained":
  supported values: "Detected", "Analyzing", "Remediating", "Resolved", "Escalated", "Failed"
unknown field spec.requiresHumanAction
unknown field spec.requiredAction
```

O fix do GAP-03 estava no binário, mas **0% efetivo em runtime** — a API server rejeitava todo update porque o CRD instalado no cluster continuava na versão anterior.

### Causa Raiz
**Comportamento padrão do Helm 3** (restrição dura, não bug):

> CRDs in the `crds/` directory are installed before any other resource on `helm install`, but they are **never** modified or deleted by `helm upgrade`.

Resultado: usuários que fazem `helm upgrade` recebem o binário novo mas mantêm o CRD antigo. Toda mudança aditiva no schema (novo enum, novo campo de spec) silenciosamente quebra o reconcile.

### Fix (3 camadas — defesa em profundidade)

**1. Pre-install/pre-upgrade Hook** (`deploy/helm/{chatcli-operator,chatcli}/templates/crd-upgrade-hook.yaml`)

Job Helm que roda ANTES do controller rollar:

- `ConfigMap` com todas as CRDs do chart (via `Files.Glob "crds/*.yaml"`)
- `ServiceAccount` + `ClusterRole` (verbs `get, list, create, update, patch` — `delete` **NÃO** concedido, hook não pode destruir CRDs)
- Job `registry.k8s.io/kubectl:v1.31.10` (imagem oficial distroless do projeto Kubernetes) que monta o ConfigMap e roda `kubectl apply --server-side --force-conflicts -f /crds`
- Annotations: `helm.sh/hook: pre-install,pre-upgrade`, `helm.sh/hook-weight: "-5"`, `helm.sh/hook-delete-policy: before-hook-creation,hook-succeeded`
- Bounded: `activeDeadlineSeconds: 300`, `backoffLimit: 2`, `ttlSecondsAfterFinished: 600`

Configuração via `values.yaml`:

```yaml
crdUpgrade:
  enabled: true             # opt-out via false (quando você gerencia CRDs via GitOps separado)
  image:
    repository: registry.k8s.io/kubectl
    # Use sempre um tag fully-qualified de patch (v1.31.10) — NÃO use major-minor
    # ("v1.31") pois o registry.k8s.io não resolve essa forma. Bitnami/kubectl
    # NÃO é recomendado: o catálogo foi descontinuado pela Broadcom em 2025 e
    # tags como "1.31" deixaram de existir, quebrando o upgrade silenciosamente.
    tag: "v1.31.10"
  resources:
    requests: { cpu: 50m, memory: 64Mi }
    limits:   { cpu: 200m, memory: 128Mi }
  tolerations: []
  nodeSelector: {}
```

**2. Floor 11 (CRD drift) estendido** (`scripts/qg/crd-drift.sh`)

O gate original só comparava `operator/api/v1alpha1/*.go` (controller-gen output) com `operator/config/crd/bases/`. A extensão também compara **cada cópia em `deploy/helm/*/crds/`** file-by-file contra o canonical. Isso pega o cenário onde o canonical está em sync com o Go mas as cópias do chart divergiram (GAP-06 class, validado na primeira execução — capturou drift legítimo do enum `MOONSHOT` que escapou em um PR anterior).

**3. Testes de paridade Go ↔ CRD** (`operator/api/v1alpha1/crd_enum_parity_test.go`)

- `TestIssueStateEnumParity`: cada constante `IssueState` deve aparecer no enum `.status.state` das 3 cópias de CRD
- `TestPostMortemSpecFieldParity`: `requiresHumanAction` e `requiredAction` devem estar no `.spec.properties` das 3 cópias do PostMortem CRD
- Falhas imprimem a receita exata de sync

### Como verificar pós-upgrade

```bash
# 1. Hook executou
kubectl -n <ns> get jobs -l app.kubernetes.io/component=crd-upgrade-hook
# STATUS COMPLETIONS DURATION
# Complete 1/1 12s

# 2. CRD tem o enum novo
kubectl get crd issues.platform.chatcli.io -o jsonpath='{.spec.versions[0].schema.openAPIV3Schema.properties.status.properties.state.enum}'
# ["Detected","Analyzing","Remediating","Contained","Resolved","Escalated","Failed"]

# 3. PostMortem CRD tem os novos campos
kubectl get crd postmortems.platform.chatcli.io -o jsonpath='{.spec.versions[0].schema.openAPIV3Schema.properties.spec.required}'
# ["issueRef","resource","severity"]   (requiresHumanAction/requiredAction são opcionais)

# 4. Logs do Job
kubectl -n <ns> logs job/$(kubectl -n <ns> get jobs -l app.kubernetes.io/component=crd-upgrade-hook -o jsonpath='{.items[0].metadata.name}')
# Applying 17 CRD(s) from chart chatcli-operator-<version>
# CRD upgrade complete
```

### Workaround para releases afetadas (`bitnami/kubectl` tag inválida)

<Warning>
O primeiro shipping do hook (release `chatcli-operator-1.122.3`) referenciou
`bitnami/kubectl:1.31` como imagem default — tag que **não existe** no Docker
Hub porque o catálogo Bitnami foi descontinuado pela Broadcom em 2025 (apenas
`latest` permanece publicado). O Job entra em `ImagePullBackOff` e o
`helm install` / `helm upgrade` trava no hook.

Se você atingir esse estado antes do próximo patch (com `registry.k8s.io/kubectl`
como default), três caminhos:
</Warning>

**A. Override do values na hora do upgrade** (recomendado):

```bash
helm upgrade chatcli-operator oci://ghcr.io/diillson/charts/chatcli-operator \
  --namespace chatcli-system --reuse-values \
  --set crdUpgrade.image.repository=registry.k8s.io/kubectl \
  --set crdUpgrade.image.tag=v1.31.10
```

**B. Desabilitar o hook e aplicar os CRDs manualmente**:

```bash
# 1. Deletar o Job travado (se já criado)
kubectl -n chatcli-system delete job -l app.kubernetes.io/component=crd-upgrade-hook

# 2. Aplicar CRDs direto do release tag
kubectl apply --server-side --force-conflicts -f \
  https://raw.githubusercontent.com/diillson/chatcli/v1.122.3/deploy/helm/chatcli-operator/crds/

# 3. Re-rodar o upgrade com hook desabilitado
helm upgrade chatcli-operator oci://ghcr.io/diillson/charts/chatcli-operator \
  --namespace chatcli-system --reuse-values \
  --set crdUpgrade.enabled=false
```

**C. Aguardar o patch release** que substitui o default por `registry.k8s.io/kubectl:v1.31.10`
(rastreado em `fix/aiops-gap-06-kubectl-image-tag`).

### Quando desabilitar o hook

Apenas quando você gerencia CRDs **fora do Helm**:

- Chart separado (padrão cert-manager): `helm install chatcli-crds <separate-chart>` antes do operator
- GitOps com aplicação explícita de CRDs (ArgoCD App-of-Apps, FluxCD `Kustomization` apontando para `crds/`)
- Operadores corporate que exigem revisão manual de CRDs pré-rollout

Nesses casos: `helm install ... --set crdUpgrade.enabled=false` ou no `values-prod.yaml`:

```yaml
crdUpgrade:
  enabled: false
```

O `NOTES.txt` printa um warning explícito com o comando manual quando o hook está desabilitado.

### Schema affected
- `values.yaml` + `values.schema.json` (ambos charts) — bloco `crdUpgrade` aditivo
- Novos templates: `crd-upgrade-hook.yaml` em ambos charts
- `scripts/qg/crd-drift.sh` — Floor 11 estendido
- `operator/api/v1alpha1/crd_enum_parity_test.go` — novo arquivo de teste

## GAP-07 — Campos `requiresHumanAction` / `requiredAction` `null` em runtime (Baixa)

### Sintoma

Após a transição do Issue para `Contained`, consultar os campos retornava `null`:

```bash
kubectl get issue chaos-crashloop-pod-not-ready-1779561258 \
  -o jsonpath='{.status.requiresHumanAction}'
# (vazio)

kubectl get postmortem pm-chaos-crashloop-... \
  -o jsonpath='{.spec.requiresHumanAction}'
# (vazio)
```

O operator **logava corretamente** a ação no startup do reconcile:

```
"Issue transitioned to Contained (requires human action)"
  required_action: "restore the deployment's replicas to the desired count after fixing the root cause"
```

Mas o valor nunca aparecia no CR — operadores tinham que `kubectl logs` no operator pra descobrir o follow-up. UX ruim em produção.

### Causa Raiz

Dois bugs ortogonais:

1. **Convenção K8s violada no PostMortem**: os campos `RequiresHumanAction` / `RequiredAction` foram adicionados em `PostMortemSpec` no GAP-03. **Convenção K8s**: `Spec` carrega entrada do usuário; `Status` carrega fatos derivados pelo controller. Esses dois campos são 100% controller-derived (nenhum usuário os preenche). Pior: o controller escrevia via `Status().Update()` enquanto o CRD declarava os campos em `Spec` — daí o `null` silencioso.

2. **Ausência no Issue**: o `IssueStatus` só registrava a condition `RequiresHumanAction: True` — nenhum field tipado. `kubectl get issue -o jsonpath='{.status.requiresHumanAction}'` retornava `null` por design.

### Fix

**Schema move (`operator/api/v1alpha1`)**:

```diff
 type PostMortemSpec struct {
   IssueRef IssueRef    `json:"issueRef"`
   Resource ResourceRef `json:"resource"`
   Severity IssueSeverity `json:"severity"`
-  RequiresHumanAction bool   `json:"requiresHumanAction,omitempty"`
-  RequiredAction      string `json:"requiredAction,omitempty"`
 }

 type PostMortemStatus struct {
   ...campos existentes...
+  RequiresHumanAction bool   `json:"requiresHumanAction,omitempty"`
+  RequiredAction      string `json:"requiredAction,omitempty"`
 }

 type IssueStatus struct {
   ...campos existentes...
+  RequiresHumanAction bool   `json:"requiresHumanAction,omitempty"`
+  RequiredAction      string `json:"requiredAction,omitempty"`
 }
```

**Writers (`operator/controllers`)**:

- `handleRemediating` (Issue → Contained): persiste `issue.Status.RequiresHumanAction = true` e `issue.Status.RequiredAction = ...` junto com as conditions existentes.
- `handleContained` (auto-resolve para Resolved): limpa os campos + adiciona `RequiresHumanAction: False` condition (a ação já foi tomada, o estado não pode mentir).
- `generatePostMortem`: lê o outcome de containment up-front e o helper de status `applyPostMortemStatusFields` persiste em `pm.Status.RequiresHumanAction` via `Status().Update()` — caminho K8s correto.

**4 readers atualizados**:

- `PostMortemReconciler` (revert do Closed indevido): lê de `pm.Status.RequiresHumanAction` (era `Spec`)
- `analytics.foldPostMortemSummary` (contador no dashboard): idem
- `handlePostMortemAckHumanAction` (endpoint REST `/ack-human-action`): valida `Status`
- `postmortemToItem` (mapper REST): lê de `Status`

**REST API** (`operator/api/rest/types.go`):

- `IncidentItem` ganha `requiredAction` (string) pra dashboards renderizarem a ação sem precisar fetch do PostMortem matching.
- `issueToIncidentItem` lê `iss.Status.RequiresHumanAction` / `RequiredAction` com fallback de compatibilidade pra Issues vindas de 1.122.x (lê condition se field vazio).

**CRDs**: regenerados via controller-gen e sincronizados nas 3 cópias.

**Tests de paridade** (`operator/api/v1alpha1/crd_enum_parity_test.go`):

- `TestPostMortemSpecFieldParity` substituído por `TestPostMortemStatusFieldParity` — também afirma que os fields **NÃO estão em Spec** (defesa em profundidade contra hand-edit que reintroduza a duplicação).
- Novo `TestIssueStatusHumanActionParity` cobre o Issue CRD nas 3 cópias.

### Verificação pós-fix

```bash
# 1. Field tipado retorna a ação direto
kubectl get issue <name> -o jsonpath='{.status.requiredAction}'
# restore the deployment's replicas to the desired count after fixing the root cause...

kubectl get postmortem pm-<name> -o jsonpath='{.status.requiredAction}'
# (mesmo texto)

# 2. Flag boolean funciona pra dashboards
kubectl get issue <name> -o jsonpath='{.status.requiresHumanAction}'
# true

# 3. REST API expõe o field direto no IncidentItem
curl $AIOPS_URL/api/v1/incidents/<name> | jq '.spec.requiresHumanAction, .spec.requiredAction'

# 4. Auto-resolve limpa os campos
kubectl get issue <name> -o jsonpath='{.status.requiresHumanAction}'
# (vazio depois que humano restaura replicas e operator transita para Resolved)
```

### Migração

`PostMortemSpec.RequiresHumanAction` / `RequiredAction` → `PostMortemStatus` é uma mudança de schema **breaking** em `v1alpha1`. Os fields foram introduzidos uma release atrás (1.122.0), então impacto pra consumidores é mínimo. O data path é controller-only (nenhum usuário-input CR jamais setou esses fields), tornando a migração invisível na prática. O hook de upgrade de CRD (GAP-06) atualiza o schema automaticamente no `helm upgrade`.

### Schema affected

- `operator/api/v1alpha1/postmortem_types.go` — move dos campos Spec → Status
- `operator/api/v1alpha1/issue_types.go` — adição em `IssueStatus`
- `operator/api/rest/types.go` — `IncidentItem.RequiredAction` adicionado
- CRDs regenerados em `operator/config/crd/bases/` + 2 cópias Helm
- `operator/controllers/issue_controller.go`, `postmortem_controller.go`, `api/rest/analytics.go`, `api/rest/server.go` — readers e writers atualizados
