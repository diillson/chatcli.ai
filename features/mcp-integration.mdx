O **MCP (Model Context Protocol)** e um protocolo aberto para interoperabilidade de ferramentas com modelos de IA. O ChatCLI integra com servidores MCP, permitindo que a IA acesse ferramentas externas como sistema de arquivos, buscas web, bancos de dados e qualquer serviĂ§o compatĂ­vel.


## Transportes

<Tabs>
  <Tab title="stdio">
    O transporte `stdio` inicia um processo local e se comunica via stdin/stdout usando JSON-RPC 2.0:

    ```json
    {
      "name": "filesystem",
      "transport": "stdio",
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-filesystem", "/workspace"]
    }
    ```

    **Uso**: Servidores locais distribuidos como pacotes npm, binarios standalone ou scripts.
  </Tab>
  <Tab title="SSE (HTTP+SSE)">
    O transporte `sse` conecta a um servidor HTTP remoto que expĂ”e **dois endpoints** â `/sse` para o stream e `/messages` para os POSTs JSON-RPC, conforme o modelo original da spec MCP:

    ```json
    {
      "name": "remote-tools",
      "transport": "sse",
      "url": "http://mcp-server:8080/sse"
    }
    ```

    O listener do GET `/sse` Ă© **supervisionado**: queda de conexĂŁo (proxy idle timeout, server restart) dispara reconnect com backoff exponencial + jitter (500 ms â 30 s cap), pending RPCs sobrevivem ao reconnect, e push notifications no mesmo stream alimentam [MCP Channels](/features/mcp-channels).

    **Uso**: servidores remotos baseados no modelo HTTP+SSE (servers anteriores Ă  spec 2025-03-26 ou que mantĂȘm esse modelo por compatibilidade).
  </Tab>
  <Tab title="http (Streamable HTTP)">
    O transporte `http` conecta a um servidor remoto que expĂ”e **um Ășnico endpoint** conforme a spec MCP **2025-03-26** (Streamable HTTP). Adotado pelo SDK oficial e por hosts como Cloudflare Workers MCP, FastMCP, mcp.deepwiki.com e gateways MCP gerenciados:

    ```json
    {
      "name": "remote-tools",
      "transport": "http",
      "url": "https://mcp-server.example.com/mcp/",
      "auth": {
        "type": "bearer",
        "token": "${MCP_TOKEN}"
      }
    }
    ```

    Cada chamada Ă© um POST Ășnico; a resposta pode chegar como JSON one-shot, como SSE upgrade no mesmo response (carregando a resposta e notificaĂ§Ă”es `notifications/*` intercaladas) ou como `202 Accepted` para notificaĂ§Ă”es clienteâservidor.     Push notifications server-initiated sĂŁo lidas por um **listener GET separado** (opt-in pela spec).     <Note>A barra final na `url` Ă© significativa. VĂĄrios servidores Streamable HTTP sĂł respondem em `/mcp/` e 307-redirecionam `/mcp`. Detalhes e troubleshooting completos: [ConfiguraĂ§ĂŁo de MCP Servers â http (Streamable HTTP)](/reference/mcp-config#http-streamable-http).</Note>

## Overrides de Built-ins (Shadow + Fallback)

Quando um servidor MCP oferece uma ferramenta que **substitui** um plugin nativo (ex: `@webfetch`, `@websearch`), use o campo `overrides` para declarar quais built-ins ele substitui. O ChatCLI **esconde automaticamente** os built-ins listados enquanto o servidor MCP estiver conectado. Se o servidor **desconectar**, os built-ins sĂŁo **restaurados imediatamente** â o usuĂĄrio nunca perde a capacidade.

### Por que usar overrides?

Sem `overrides`, o LLM vĂȘ duas ferramentas que fazem a mesma coisa (ex: `@webfetch` e `mcp_web_fetch`) e escolhe aleatoriamente. Com `overrides`, o comportamento Ă© **determinĂ­stico**: o LLM sĂł vĂȘ a ferramenta MCP, sem ambiguidade.

### ConfiguraĂ§ĂŁo

Adicione `overrides` na configuraĂ§ĂŁo do servidor MCP:

<Tabs>
  <Tab title="CLI (mcp_servers.json)">
    ```json
    {
      "mcpServers": [
        {
          "name": "web-tools",
          "transport": "sse",
          "url": "http://mcp-web:8080/sse",
          "enabled": true,
          "overrides": ["@webfetch", "@websearch"]
        }
      ]
    }
    ```
  </Tab>
  <Tab title="Helm (values.yaml)">
    ```yaml
    mcp:
      enabled: true
      servers:
        - name: web-tools
          transport: sse
          url: "http://mcp-web:8080/sse"
          enabled: true
          overrides: ["@webfetch", "@websearch"]
    ```
  </Tab>
  <Tab title="Operator (Instance CRD)">
    ```yaml
    apiVersion: platform.chatcli.dev/v1alpha1
    kind: Instance
    metadata:
      name: my-chatcli
    spec:
      mcp:
        enabled: true
        servers:
          - name: web-tools
            transport: sse
            url: "http://mcp-web:8080/sse"
            enabled: true
            overrides: ["@webfetch", "@websearch"]
    ```
  </Tab>
</Tabs>

### Como funciona

```text
MCP server "web-tools" CONECTADO + overrides: ["@webfetch", "@websearch"]
  LLM vĂȘ:     mcp_web_fetch, mcp_web_search, @coder, ...
  LLM NĂO vĂȘ: @webfetch, @websearch  (escondidos)
```

<Note>
O sync de shadow acontece **a cada turno de conversa**, nĂŁo apenas no startup. Isso significa que se o MCP server cair no meio de uma conversa, os built-ins voltam **na prĂłxima mensagem** â sem restart necessĂĄrio.

### Regras importantes

| Regra | Detalhe |
|:------|:--------|
| **Nomes exatos** | Use o nome completo do built-in, incluindo o `@` (ex: `"@webfetch"`, nĂŁo `"webfetch"`) |
| **MĂșltiplos servers** | VĂĄrios servers podem declarar overrides diferentes. Se dois servers fazem override do mesmo built-in, basta **um** estar conectado para manter o shadow |
| **Server desabilitado** | `"enabled": false` no server = overrides **nĂŁo** se aplicam (server nunca conecta) |
| **Override parcial** | VocĂȘ pode fazer override de apenas um built-in (ex: sĂł `@websearch`) e manter os outros visĂ­veis |
| **Sem efeito no @coder** | O `@coder` nĂŁo pode ser overridden via MCP â suas subcommands sĂŁo tools nativas do modo /coder |

<Warning>
O campo `overrides` nĂŁo desinstala nem modifica o plugin built-in. Ele apenas o **esconde do prompt** do LLM. O plugin continua disponĂ­vel internamente e volta automaticamente quando necessĂĄrio. Isso Ă© **shadow, nĂŁo delete**.

## Push notifications â MCP Channels

AlĂ©m de tools (request/response), o ChatCLI aceita **push notifications** de qualquer servidor MCP nos trĂȘs transports. Notifications sĂŁo mensagens JSON-RPC **sem `id`** (a forma que a spec reserva para "o server quer me dizer algo, nĂŁo pediu nada em troca"). O ChatCLI captura, persiste e injeta automaticamente nas prĂłximas conversas, e â opcionalmente â aciona o agent via rules de trigger.

### O essencial

- **Funciona em `stdio`, `sse` e `http`** â cada transport tem seu prĂłprio listener:
  - `sse`: o GET `/sse` que jĂĄ carrega tool responses recebe notifications no mesmo stream.
  - `http`: um GET separado no endpoint (opt-in pelo cliente, opcional pela spec) puxa notifications. Servidores que nĂŁo suportam respondem 405/404/501 e o listener para limpo, sem retry.
  - `stdio`: linhas JSON-RPC no stdout sem `id` sĂŁo tratadas como notifications.
- **PersistĂȘncia durĂĄvel** em `~/.chatcli/mcp/channels.jsonl` (atĂ© 10 MiB, rotaÂ­ciona, recarrega no boot).
- **Auto-injection** das 5 mais recentes no system prompt de `chat`/`agent`/`coder` (uncached pra nĂŁo trashar cache).
- **Filtro per-server**: campo `channels` em `mcp_servers.json` aceita allow-list literal.
- **Rules engine opcional** em `~/.chatcli/mcp/triggers.json` com 3 modos: `notify` (banner discreto, default), `confirm` (yes/no manual), `auto` (executa o agent â requer whitelist de tools).
- **Slash commands**: `/channel list`, `/channel ack`, `/channel pause`, `/channel rules`, `/channel confirm <id>`, `/channel run <seq>`, `/channel inject`.
