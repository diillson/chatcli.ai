ChatCLI synthesizes speech via the **`llm/tts`** package (mirroring `llm/transcription`): the **`@speak`** tool produces an audio file, and the gateway can **reply with voice**. **Local/keyless-first**, and **not tied to one provider**.

---

## Backend selection (local first)

`tts.NewFromEnv` picks in this order, degrading to `null` (disabled) when nothing is configured:

1. `CHATCLI_TTS_CMD` â†’ a local command (template with `{text}` and `{output}`). Keyless.
2. `CHATCLI_TTS_URL` â†’ an OpenAI-compatible endpoint (`/audio/speech`), keyless unless `CHATCLI_TTS_KEY`.
3. Local CLI on PATH: macOS `say`, `espeak-ng`, `espeak` â€” used automatically with zero config.
4. `OPENAI_API_KEY` â†’ OpenAI TTS.
5. `GROQ_API_KEY` â†’ Groq TTS (same OpenAI shape).
6. `GOOGLEAI_API_KEY`/`GEMINI_API_KEY` â†’ **native Gemini TTS** (returns PCM, wrapped in WAV).

`CHATCLI_TTS_PROVIDER` pins a backend (`command|url|openai|groq|google`). `CHATCLI_TTS_VOICE` and `CHATCLI_TTS_MODEL` tune voice/model.

<Info>
Genuinely multi-provider: any server speaking `/audio/speech` works by pointing `CHATCLI_TTS_URL` at it. Anthropic and most of the 14 chat providers do **not** expose a speech API â€” that's why it can't be "extended to all": the endpoint doesn't exist.
</Info>

## Gateway voice replies

With `CHATCLI_GATEWAY_VOICE_REPLY=true`, the runner synthesizes the final answer and attaches the audio. The **Telegram** adapter delivers it as a native **voice note** (ogg/opus via `sendVoice`, otherwise `sendAudio`), with the text as caption, and **falls back to text** on any failure. Adapters without audio support keep sending text â€” a reply is never lost.

```bash
export CHATCLI_GATEWAY_VOICE_REPLY=true
export CHATCLI_TTS_VOICE_FORMAT=ogg   # native voice note on Telegram
```

Check the state in `/config` (*Integrations Â· Gateway Â· Voice reply (TTS)*).
