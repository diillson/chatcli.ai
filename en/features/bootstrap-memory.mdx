ChatCLI offers two complementary systems for customizing and contextualizing the agent: **bootstrap files** to define personality and rules, and **persistent memory** to maintain context across sessions.

---

## Bootstrap Files

Bootstrap files are Markdown documents automatically loaded into the agent's system prompt. They define who the assistant is, how it behaves, and which rules it should follow.

### Supported Files

| File | Purpose | Example |
|:--------|:----------|:--------|
| `SOUL.md` | Assistant personality and tone | "You are a pragmatic and direct DevOps assistant" |
| `USER.md` | User preferences and context | "The user prefers Go, uses Kubernetes, works with microservices" |
| `IDENTITY.md` | Agent identity and capabilities | "You are ChatCLI, a terminal assistant with access to tools" |
| `RULES.md` | Rules and restrictions | "Never execute destructive commands without confirmation" |
| `AGENTS.md` | Sub-agent definitions | "Agent @coder: code specialist..." |

### Loading Priority

Files are searched at two levels, with the workspace taking priority:

<Steps>
  <Step title="Workspace (./)">
    Project-specific configurations. Takes **priority** over global.
  </Step>
  <Step title="Global (~/.chatcli/)">
    User default configurations. Serves as a **fallback** when the file does not exist in the workspace.
  </Step>
</Steps>

If the same file exists at both levels, the workspace version prevails. Global files serve as fallback.

### Example

<Tabs>
  <Tab title="SOUL.md (Global)">
    Create `~/.chatcli/SOUL.md`:

    ```markdown
    # Personality

    You are a technical assistant specialized in software engineering.
    Be concise and direct. Prefer practical examples over theoretical explanations.
    When suggesting code, use best practices and tests.

    # Tone

    - Professional but approachable
    - Prefer short and objective responses
    - Use bullet points for lists
    ```
  </Tab>
  <Tab title="USER.md (Project)">
    Create `./USER.md` in your project:

    ```markdown
    # Project Context

    - Stack: Go 1.25, gRPC, Kubernetes
    - Database: PostgreSQL 16
    - CI/CD: GitHub Actions
    - Style: conventional commits, trunk-based development
    ```
  </Tab>
</Tabs>

### Smart Cache

Bootstrap files use mtime-based (modification time) caching:

- On the first read, the content is cached in memory
- Subsequent reads check if the mtime has changed
- If the file was modified, the cache is automatically invalidated
- `IsStale()` checks if any file has changed since the last load

## Persistent Memory

The memory system maintains context across ChatCLI sessions, allowing the agent to "remember" facts, decisions, and patterns over time.

### Structure

```text
~/.chatcli/memory/
|-- MEMORY.md              # Long-term facts
+-- daily/
    |-- 202603/
    |   |-- 20260301.md    # Notes from March 1st
    |   |-- 20260302.md    # Notes from March 2nd
    |   +-- 20260304.md    # Today's notes
    +-- 202602/
        +-- 20260228.md    # February notes
```

### MEMORY.md — Long-Term Memory

`MEMORY.md` stores persistent facts that the agent should always consider:

```markdown
# Architectural Decisions

- We use go-prompt v0.2.6 for interactive TUI
- Charmbracelet (glamour, lipgloss) only for styling, not for TUI
- Decoupled message bus via cli/bus for agent I/O

# Project Patterns

- json.Compact takes *bytes.Buffer, not *strings.Builder
- embed.FS always uses / in paths, never filepath.Join
- Stateful parsers > regex for XML with special attributes
```

### Daily Notes

Daily notes are organized by year-month (`YYYYMM/`) and named by date (`YYYYMMDD.md`):

```markdown
# 2026-03-04

## Work Done
- Implemented fallback chain for LLM providers
- Added MCP support via stdio/SSE
- 47 files changed, +4920 lines

## Decisions
- Use init() auto-registration for providers in the registry
- cache_control:ephemeral for KV cache optimization on Anthropic
```

### Available Methods

```go
ms := workspace.NewMemoryStore(dir, logger)

// Long-term memory
ms.WriteLongTerm("# Key Facts\n- Go project")
ms.AppendLongTerm("\n- Uses gRPC for remote access")
content := ms.ReadLongTerm()

// Daily notes
ms.WriteDailyNote("Implemented feature X")
path := ms.TodayNotePath()
notes := ms.GetRecentDailyNotes(3) // last 3 notes

// Complete context for system prompt
ctx := ms.GetMemoryContext()
// Returns formatted section with "Long-term Memory" + "Daily Notes"
```

## Context Builder

The `ContextBuilder` combines bootstrap + memory into a single system prompt prefix:

```go
cb := workspace.NewContextBuilder(bootstrapDir, globalDir, memoryDir, logger)
systemPrefix := cb.BuildContext()
// Returns: bootstrap content + memory context, ready to inject into the prompt
```

The result is cached and automatically invalidated when any source file changes.

## Configuration

<Tabs>
  <Tab title="Environment Variables">
    ```bash
    CHATCLI_BOOTSTRAP_ENABLED=true
    CHATCLI_BOOTSTRAP_DIR=/path/to/bootstrap/files
    CHATCLI_MEMORY_ENABLED=true
    ```
  </Tab>
  <Tab title="Via Helm Chart">
    ```yaml
    # values.yaml
    bootstrap:
      enabled: true
      definitions:
        SOUL.md: |
          You are a DevOps assistant...
        USER.md: |
          The user prefers Go...

    memory:
      enabled: true
      # Uses the persistence PVC by default
    ```

    The Helm chart creates ConfigMaps for bootstrap files and mounts them at `/home/chatcli/.chatcli/bootstrap/`. Memory uses the sessions PVC for persistence.
  </Tab>
</Tabs>

## Best Practices

<CardGroup cols={2}>
  <Card title="Global SOUL.md, per-project USER.md" icon="layer-group">
    Keep your preferred personality globally and technical context per project.
  </Card>
  <Card title="Keep MEMORY.md concise" icon="compress">
    Keep only stable and confirmed facts — not session-specific ones.
  </Card>
  <Card title="Daily notes for journaling" icon="calendar-day">
    Use them to record decisions, solved problems, and temporal context.
  </Card>
  <Card title="Don't duplicate CLAUDE.md" icon="copy">
    If you already use CLAUDE.md or project instructions, avoid duplicating them in bootstrap.
  </Card>
</CardGroup>

<Tip>Periodically review your memories and remove outdated ones to keep the context relevant.</Tip>
