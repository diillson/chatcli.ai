The `mcp_servers.json` file defines which MCP servers ChatCLI should connect to. It is loaded automatically at session startup.

---

## Location

| Location | Priority | Description |
|:---------|:---------|:-----------|
| `~/.chatcli/mcp_servers.json` | Default | User's global configuration |
| `$CHATCLI_MCP_CONFIG` | Override | Custom path via environment variable |
| `--mcp-config` flag | Highest | Override via server flag |

<Info>In client mode (TTY), simply create the file at `~/.chatcli/mcp_servers.json` — ChatCLI auto-detects and initializes servers without needing environment variables.</Info>

---

## Format

```json
{
  "mcpServers": [
    {
      "name": "string",           // Unique server name
      "transport": "stdio|sse",   // Transport type
      "command": "string",        // Command to start (stdio)
      "args": ["string"],         // Command arguments (stdio)
      "env": {                    // Environment variables (stdio)
        "KEY": "VALUE"
      },
      "url": "string",            // Server URL (sse)
      "enabled": true,            // Enable/disable without removing
      "overrides": ["string"]     // Built-ins this server replaces
    }
  ]
}
```

---

## Fields

| Field | Type | Required | Description |
|:------|:-----|:---------|:-----------|
| `name` | string | ✅ | Unique server identifier. Used in logs and `[MCP:<name>]` tool prefix |
| `transport` | string | ✅ | Transport type: `"stdio"` (local) or `"sse"` (remote) |
| `command` | string | stdio ✅ | Command to start the MCP process |
| `args` | string[] | ❌ | Arguments passed to the command |
| `env` | object | ❌ | Process environment variables (key-value) |
| `url` | string | sse ✅ | SSE endpoint URL for the remote server |
| `enabled` | bool | ❌ | Default: `false`. Disable without removing from config |
| `overrides` | string[] | ❌ | Names of built-in plugins this server replaces. When connected, the listed built-ins are **hidden** from the LLM prompt. If the server disconnects, they are **automatically restored**. E.g.: `["@webfetch", "@websearch"]` |

## Examples by Transport

<Tabs>
  <Tab title="stdio (Local)">
    Local servers communicating via stdin/stdout with JSON-RPC 2.0:

    ```json
    {
      "mcpServers": [
        {
          "name": "filesystem",
          "transport": "stdio",
          "command": "npx",
          "args": ["-y", "@anthropic/mcp-server-filesystem", "/home/user/projects"],
          "enabled": true
        }
      ]
    }
    ```

    <Tip>ChatCLI manages the process lifecycle: starts on init, kills on shutdown. Content-Length framing (LSP-style) is used for communication.</Tip>
  </Tab>
  <Tab title="SSE (Remote)">
    Remote servers via Server-Sent Events:

    ```json
    {
      "mcpServers": [
        {
          "name": "remote-tools",
          "transport": "sse",
          "url": "https://mcp-server.example.com/sse",
          "enabled": true
        }
      ]
    }
    ```

    The SSE handshake works as follows:
    1. GET `/sse` → receives `endpoint` event with messages URL
    2. POST JSON-RPC messages to the discovered URL
    3. Responses arrive via SSE stream
  </Tab>
</Tabs>

## Full Examples

### Multiple Servers

```json
{
  "mcpServers": [
    {
      "name": "filesystem",
      "transport": "stdio",
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-filesystem", "/workspace"],
      "enabled": true
    },
    {
      "name": "github",
      "transport": "stdio",
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-github"],
      "env": {
        "GITHUB_TOKEN": "ghp_xxxxxxxxxxxxxxxxxxxx"
      },
      "enabled": true
    },
    {
      "name": "postgres",
      "transport": "stdio",
      "command": "/usr/local/bin/mcp-postgres",
      "args": ["--connection-string", "postgresql://localhost:5432/mydb"],
      "env": {
        "PGPASSWORD": "secret"
      },
      "enabled": true
    },
    {
      "name": "web-search",
      "transport": "sse",
      "url": "http://localhost:8080/sse",
      "enabled": true,
      "overrides": ["@webfetch", "@websearch"]
    },
    {
      "name": "slack",
      "transport": "stdio",
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-slack"],
      "env": {
        "SLACK_BOT_TOKEN": "xoxb-xxxxxxxxxxxxxxxxxxxx"
      },
      "enabled": false
    }
  ]
}
```

<Warning>Never commit tokens or passwords in the config file. Use system environment variables or a secrets manager.</Warning>

## Protocol

ChatCLI implements **MCP Protocol v2024-11-05** with:

| Method | Direction | Description |
|:-------|:---------|:-----------|
| `initialize` | Client → Server | Initial handshake with version and capabilities |
| `notifications/initialized` | Client → Server | Confirms initialization |
| `tools/list` | Client → Server | Discovers available tools |
| `tools/call` | Client → Server | Executes a tool with arguments |

### Tool Naming

```text
Original name on server:  read_file
Name in ChatCLI:          mcp_read_file
Description:              [MCP:filesystem] Reads a file from the filesystem
```

<Info>The `mcp_` prefix is added automatically to avoid collisions with native tools (plugins, @coder, @webfetch, etc.).</Info>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Server won't connect">
    Check:
    1. The command exists and is in PATH (`which npx`)
    2. Config file is at `~/.chatcli/mcp_servers.json`
    3. `enabled` is `true`
    4. Use `/mcp status` to see connection errors
    5. Check logs with `CHATCLI_LOG_LEVEL=debug`
  </Accordion>
  <Accordion title="Tools not showing up">
    1. The server may not support `tools/list`
    2. Use `/mcp tools` to list discovered tools
    3. Restart with `/mcp restart`
  </Accordion>
  <Accordion title="Tool execution timeout">
    Default timeout is 30s per call. If the server takes longer:
    1. Check server health
    2. Consider using SSE for slow operations
  </Accordion>
  <Accordion title="Override not working — LLM still sees the built-in">
    Check:
    1. The name in `overrides` **exactly** matches the plugin name, including the `@` (e.g., `"@webfetch"`, **not** `"webfetch"`)
    2. The MCP server is **connected** (`/mcp status`)
    3. The server's `enabled` field is `true`
    4. If the server crashed and reconnected, the override resumes automatically on the **next conversation turn**
  </Accordion>
  <Accordion title="Built-in disappeared but the MCP server has no equivalent tool">
    The `overrides` field **only hides** the built-in — it does not validate whether the MCP server actually provides an equivalent tool. If you set `"overrides": ["@webfetch"]` but the MCP server has no `web_fetch` tool, the LLM simply won't have that capability. Remove the override or add the tool to the MCP server.
  </Accordion>
  <Accordion title="I want to force the built-in even with MCP active">
    Remove the built-in from the MCP server's `overrides` list. Both tools (MCP and built-in) will be visible simultaneously and the LLM will choose based on each tool's name and description.
  </Accordion>
</AccordionGroup>
